# molding-optima 项目重构方案

## 一、目标与边界

**目标**：在 molding-optima 项目原地按 molding-expert 的最新设计进行重构，实现以下升级：
1. 后端：Django 5.2 + 复用 molding-expert 的脚手架（`_moldx`、`extensions`、`utils`）
2. 后端：复用 `identity`、`masterdata` 两个 app；按 molding-optima 业务**重写一个智能工艺专用的 process app**；**复用** `filecenter`、`bootstrap`，并**精简引入** `reporting`
3. 前端：当前保留 Vue 2 + Element UI（选项式 API），沿用 molding-expert 的页面与组件设计，**保留原有模块划分**全量迁移 molding-optima 原有页面；待整体项目稳定后再统一升级到 Vue 3 + Vite 5 + TypeScript + Element Plus

**项目背景**：
- **molding-expert**：当前最新的设计（从模型到页面），是本次重构的设计源头与脚手架/通用模块复用源
- **molding-optima**：历史上从老系统中切出的分支，独立发展了一段时间，功能可用但整体设计已不合适，本次需按 molding-expert 的新设计重塑

**业务边界**（不引入）：
- 不引入 molding-expert 的 moldflow、schedule、trial、inventory、costing、notice、monitoring
- 不引入 channels/websocket（除非后续明确需要）
- 不引入 MongoDB/MQTT/Redis/Celery（保持与 molding-expert 主线对齐：MySQL 单库）

**模块引入策略**：
- `filecenter`：直接复用 molding-expert 版本，负责文件上传/下载/存储
- `reporting`：复用结构（models、views、services、urls 分层），但**services 按 molding-optima 实际业务精简**（仅保留工艺相关的导出/报告）
- `bootstrap`：复用结构与初始化脚本，**按 molding-optima 业务裁剪**（仅保留 identity 初始化、组织初始化、字典数据，移除试模/模流/排程相关的 fixtures）

**保留策略**：
- 保留：`backend/`（已按 molding-expert 最新设计完成重构）、`frontend/`（Vue 2 + Element UI 基础环境已搭好，沿用 molding-expert 设计）、`old/`（历史参考）、`prototype/`（页面原型参考）、`.gitignore`、`README.md`、`requirements.txt`
- 保留 `molding-optima/backend/` 内已有的 `identity/models/`、`masterdata/services/` 等近期修改的代码（已从 molding-expert 迁移）

**重构分阶段策略**：
本方案仅覆盖**基础框架重构**（第一阶段），主要产出：
- 可运行的 Django 脚手架 + Vue 2 + Element UI 前端
- identity、masterdata、process、search、filecenter、reporting、bootstrap 7 个业务 app 的基础 CRUD
- 原 molding-optima 的页面按 molding-expert 的 Vue 设计语言迁移
- 统一的 API 调用、权限、响应处理架构

**当前进度**：
- 后端 7 个 app 已完成（`_moldx`、`extensions`、`utils`、`identity`、`masterdata`、`search`、`filecenter`、`reporting`、`bootstrap`、`process` 重写）
- 前端基础环境（Vue 2 + Element UI + Vuex）已搭好
- 处于审核阶段

**第二阶段（后续迭代，本方案不包含）**：
基础框架稳定后，从 `old/` 目录提炼**核心智能优化业务逻辑**导入新框架，主要参考：
- `old/mdprocess/`（工艺优化、规则、专家调优核心业务逻辑）
- `old/hsmolding/views/process_optimize_views.py`、`process_index_views.py`（工艺索引与优化）
- `old/hsmolding/services/process_service.py`（工艺服务）
- `old/mdprocess/rule_*`（规则引擎、关键词）
- `old/mdprocess/process_expert_*`（专家调优算法）
- `old/hsmolding/services/machine_adaption_service.py`（机台适配算法）
- `old/hsmolding/services/polymer_service.py`（材料推荐算法）

**迁移原则**：以新框架的架构为准（BaseView + Pydantic Schema + service 拆分），将 old/ 中的核心算法逻辑**重写**后迁入，不直接拷贝老代码。

**注释规范**：重构期间不需在代码中标记过多"重构"信息（如「TODO 重构」「原 molding-optima 字段」「从 molding-expert 迁移」之类的注解），按正常项目开发的注释风格即可。

## 二、技术栈版本

### 后端
- Python：3.11（稳定 + Django 5.2 官方推荐）
- Django：5.2.x（最新 LTS 候选；molding-expert 与 molding-optima 当前都已用 5.2）
- MySQL：8.0（utf8mb4）
- Pydantic：v2.x（与 molding-expert 一致）
- 依赖管理：`requirements.txt`

### 前端（当前）
- Node：20.x LTS
- Vue：2.x（沿用 molding-expert 当前设计）
- UI：Element UI（沿用 molding-expert 页面与组件，避免升级带来的差异）
- 构建工具：webpack（molding-expert 当前使用）
- HTTP：axios
- 工具库：dayjs（替代 dateformat）、file-saver、xlsx、echarts
- 状态管理：Vuex（沿用 molding-expert 当前设计）

**注**：整体项目稳定后，再考虑统一升级到 Vue 3 + Vite 5 + TypeScript + Element Plus + Pinia。

## 五、关键实施步骤

### 阶段 1：后端基础设施搭建（_moldx + extensions + utils）—— ✅ 已完成

1. **复制 `_moldx/`**：从 molding-expert 复制，调整：
   - `settings.py`：精简 INSTALLED_APPS 为 `identity`、`masterdata`、`process`、`search`、`filecenter`、`reporting`、`bootstrap`
   - `settings.py`：移除 MongoDB/MQTT/飞书通知/EMAN 同步配置，保留 MySQL 单库；filecenter 需要 FILE_UPLOAD 相关设置、MAX_FILE_SIZE 等从 molding-expert 复用
   - `urls.py`：注册 7 个 app
   - `db_router.py`：简化为单 MySQL 路由
2. **复制 `extensions/`**：直接复制 molding-expert 版本，包含 BaseModel/BaseView/ApiMiddleware/Pydantic Schema
3. **复制 `utils/`**：直接复制
4. **更新 `requirements.txt`**：参考 molding-expert 的依赖，加入 Pydantic、django-cors-headers、openpyxl（reporting 导出用）、Pillow（filecenter 图片处理）等

### 阶段 2：后端业务模块 —— ✅ 已完成

1. **复制 `identity/`** 和 **`masterdata/`**：直接从 molding-expert 拷贝
2. **复制 `search/`**：直接复制
3. **复制 `filecenter/`**：直接复制 molding-expert 版本（models/services/views/urls 全部拷贝），负责处理工艺附件、模具图纸、材料证书等文件上传下载
4. **精简引入 `reporting/`**：
   - `models.py`、`schemas.py`、`urls.py`：复制自 molding-expert
   - `views/`：仅保留与 molding-optima 业务相关的视图入口（如工艺参数导出接口）
   - `services/`：**不复制** molding-expert 的 service 套件，根据 molding-optima 实际需求**新写**：
     - `process_export_service.py`：工艺参数 Excel 导出（基于 openpyxl）
     - `process_report_service.py`：工艺报告生成（可选 PDF）
   - 业务边界：仅服务于 process app，不集成试模/模流相关的报告
5. **裁剪引入 `bootstrap/`**：
   - 复制目录结构（data/、management/commands/、migrations/、scripts/）
   - `data/` 中的 fixtures：**仅保留** identity 初始账号、组织结构、字典数据；**删除**试模/模流/排程/库存相关的初始化数据
   - `management/commands/`：保留通用命令（init_data、seed_company），删除与 molding-optima 业务无关的命令
6. **重写 `process/`**：
   - `models.py`：保留 molding-optima 现有的 `ProcessCondition` 和 `ProcessParameter`（字段语义不变），继承 `BusinessBaseModel`，确保软删除、追踪、租户隔离字段齐全
   - `schemas.py`：用 Pydantic 重写 `ProcessConditionSchema`、`ProcessParameterSchema`
   - `services/process_service.py`：参考 molding-expert 的 service 拆分模式（class-based service），实现工艺参数的 CRUD、批量删除、参数移植；文件附件走 filecenter（通过文件服务 API 上传）
   - `views/processes.py`：使用 `BaseView` + `validate_parameters` 装饰器
   - `urls.py`：参考 molding-expert 的 URL 命名风格（`processes/parameter/...`）
7. **数据库初始化**：`makemigrations` + `migrate` + `bootstrap` 的 loaddata 初始化脚本
8. **验证**：启动 `runserver`，curl `/api/processes/parameter/` 与 `/api/filecenter/files/`

### 阶段 3：前端基础设施搭建 —— ✅ 已完成（审核中）

1. **初始化 Vue 2 + Element UI 项目**：保留项目名 `molding-optima`，沿用 molding-expert 的 webpack / vue.config.js
2. **集成 Element UI**：完整引入 + 中文 locale，保持与 molding-expert 版本一致
3. **集成 Vuex 状态管理**

### 阶段 4：前端按模块迁移（按依赖顺序）

迁移顺序（从底向上）：
1. `login/` + `404.vue` + `Layout`（先跑通登录跳转）
2. `superManage/`（公司、组织、角色、用户管理，统一在 `/admin/*` 下）—— identity 模块对应

**迁移规则**：
- 保留原文件路径和命名（驼峰或下划线按原文件）

- 保留原 API 端点，axios 配置新增 baseURL、token 注入、统一异常处理
